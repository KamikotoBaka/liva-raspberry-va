Modul               Tool/Lib                        Beschreibung
STT                 Whisper (OpenAI, lokal)         Sprache → Text, läuft komplett offline
NLU / Intent        Regex + eigene Logik            "Fehler identifizieren" → Intent erkennen
TTS                 Piper oder pyttsx3              Text → Sprache, lokal
REST Adapter        httpx / requests                REST-Geräte ansprechen
MQTT Adapter        paho-mqtt                       MQTT-Geräte ansprechen 
Command Dispatcher  eigene Python-Klasse            verteilt Intent → richtiger Adapter


Ablauf-Beispiel
1. User spricht: "Fehler identifizieren"
         ↓
2. STT (Whisper) → Text: "Fehler identifizieren"
         ↓
3. NLU / Intent Detection
   → Intent: identify_error
   → Entity: kein spezifisches Gerät → alle prüfen
         ↓
4. Command Dispatcher
   → ruft REST Adapter auf: GET /api/devices/status
   → ruft MQTT Adapter auf: subscribe devices/+/error
         ↓
5. Geräteantwort auswerten
   → Gerät "Lampe Wohnzimmer": status=error
         ↓
6. Antwort-Text generieren:
   → "Gerät Lampe Wohnzimmer meldet einen Fehler: Verbindung unterbrochen"
         ↓
7. TTS (Piper) → Sprachausgabe
         ↓
8. UI: alle 3 Felder aktualisiert


Überlegung:
Whisper ist zu stark für kleine geräte, die möglichkeit ist die Vosk zu nutzen oder faster-whisper 
(4x schneller als original Whisper, gleiche Qualität!), Reaktion 1-2 Sekunden.

LIVA Wakeword Training Toolkit

Added helper scripts under `training/liva`:
- `setup_train_env.ps1` creates a dedicated Python 3.11 training environment
- `prepare_dataset.py` validates and splits WAV clips
- `train_liva_verifier.py` trains a verifier model (`models/liva_verifier.pkl`)

subprocess wichtig: 
import subprocess

def execute_system_command(command: list) -> dict:
    try:
        result = subprocess.run(
            command,
            capture_output=True,
            text=True,
            timeout=10
        )
        return {
            "success": result.returncode == 0,
            "output": result.stdout,
            "error": result.stderr,
            "returncode": result.returncode
        }
    except subprocess.TimeoutExpired:
        return {"success": False, "error": "Timeout"}

device.yaml erwitern: 

intents:
  - name: "Apache neustarten"

    actions:
      - type: SHELL

    response: "Apache Server wurde neugestartet"

  - name: "System Status"
    intent: system_status  
    actions:
      - type: SHELL
        command: ["systemctl", "status", "apache2"]

!WIchtig eine white list ertellen!

ALLOWED_COMMANDS = {
    "restart_apache":  ["systemctl", "restart", "apache2"],
    "stop_apache":     ["systemctl", "stop",    "apache2"],
    "restart_nginx":   ["systemctl", "restart", "nginx"],
    "system_status":   ["systemctl", "status",  "apache2"],
    "check_disk":      ["df", "-h"],
    "check_memory":    ["free", "-h"],
}

def execute_intent_command(intent: str) -> dict:
    if intent not in ALLOWED_COMMANDS:
        return {"success": False, "error": "Befehl nicht erlaubt"}
    
    command = ALLOWED_COMMANDS[intent]
    return execute_system_command(command)

How to make switching actually affect your assistant behavior

Change processing path from typed text to audio STT.
Add an audio endpoint (for example /api/process-audio) in main.py:
receive uploaded wav/webm
convert to wav if needed
call stt.transcribe_audio(...)
then pass transcript to dispatcher.
Keep current /api/process for debug/manual text input.
That is the key point: model switching is already implemented correctly, 
but it only becomes meaningful when requests go through transcribe_audio, not transcribe_text.
